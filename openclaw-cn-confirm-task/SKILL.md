# OpenClaw 任务理解确认

这个 skill 对应 `superpowers` 里的 brainstorming 阶段，但这里不是做软件设计，而是把任务理解、边界、执行方式和完成标准对齐。

你的目标不是马上执行，而是确保你接下来执行的事情，和用户真正要的事情一致。

<HARD-GATE>
在用户确认之前，不得进入正式执行，不得调用会改变外部状态的动作，不得把“分析”伪装成“已经开始做了”。

如果任务复杂、代价高、需要多轮工具调用，不能跳过这个阶段。
</HARD-GATE>

## 你必须完成的清单

你必须按顺序完成以下事项：

1. 输出可见理解稿
2. 给出少量关键选项
3. 写出执行合同
4. 写出步骤清单
5. 请求一次性确认
6. 用户确认后，进入 `openclaw-cn-execute-plan`

## 第一步：输出可见理解稿

先用用户能快速核对的语言写出：

- `我的理解`
- `你真正要的结果`
- `我当前的关键假设`
- `最容易跑偏的地方`

不要暴露长篇思维链。只写用户检查任务方向所需的最小必要内容。

如果任务很简单，这部分可以只有几行。

如果任务复杂，这部分必须先单独讲清楚。

## 第二步：给出少量关键选项

不要让用户从零设计流程。你要主动提出 2 到 3 个真正影响结果的关键选项。

可用的选项类型包括：

- 输出风格：简洁 / 平衡 / 详细
- 执行策略：稳妥 / 标准 / 速度优先
- 交付形式：直接结果 / 表格 / 文件 / 先方案后执行
- 优先目标：准确 / 完整 / 省时间

如果某项默认值明显更合理，要明确写出推荐项。

不要问无关问题。不要把确认阶段变成问卷。

## 第三步：写出执行合同

必须显式写出：

- `Goal`
- `Deliverable`
- `Inputs`
- `Constraints`
- `Non-goals`
- `Completion gate`

要求如下：

- `Goal` 只能有一个主目标
- `Deliverable` 必须可观察
- `Constraints` 只保留硬约束
- `Non-goals` 要写清不做什么
- `Completion gate` 必须能验证，不能模糊

## 第四步：写出步骤清单

把任务拆成 `1..N` 个必要步骤。

每一步都必须：

- 足够具体
- 可以判断完成与否
- 尽量只承担一个动作目的

每步初始状态必须是以下三者之一：

- `pending`
- `in_progress`
- `done`

如果存在分支条件，必须在这里提前写出，不要等执行时再现编。

## 第五步：请求一次性确认

确认语要简短、直接。推荐类似：

`如果以上理解和方案正确，我将按这个步骤清单持续执行到完成，中间不再为普通步骤逐一请求批准。`

用户此时只需要做两件事之一：

- 确认
- 指出要修改的地方

不要让用户重新从头写流程。

## 推荐输出结构

建议使用以下结构：

- `我的理解`
- `关键选择`
- `执行合同`
- `步骤清单`
- `请确认或修改`

## 红旗信号

以下情况说明你还没确认清楚，不该进入执行：

- 你还说不清最终交付物是什么
- 你还不确定哪些约束是硬性的
- 你还不能列出完整必要步骤
- 用户看完后明显觉得“你理解偏了”

出现这些情况时，继续留在本阶段修正，不要跳到执行。
