# OpenClaw 中文复杂任务工作流

这是一个单包 workflow skill，用于把原本拆开的四段式流程合并成一个可分发版本。

它的目标是解决中文模型在复杂任务中常见的四个问题：

- 还没理解清楚就开始执行
- 多轮任务做到一半就提前结束
- 步骤很多但没有持续跟踪
- 没有真正完成就开始收尾

你必须把这个 skill 当成一个完整流程，而不是普通提示词模板。

## 触发规则

只有复杂任务才触发这个 skill。

以下情况通常应触发：

- 任务通常需要 3 步及以上才能完成
- 需要工具调用、外部动作或跨回合持续推进
- 用户目标明确，但实现路径不明确
- 输出格式、约束、边界条件较多
- 中途容易误判“已经完成”
- 做错的返工成本较高

以下情况通常不应触发：

- 简单事实问答
- 简单翻译
- 简单润色
- 轻量总结
- 单轮、单步、目标清晰的任务
- 不需要计划、不需要状态跟踪、不需要完成前验收的回答

如果犹豫是否触发，先问自己三个问题：

1. 不列步骤会不会大概率漏做？
2. 不确认理解会不会大概率跑偏？
3. 不做结束验收会不会大概率提前收尾？

如果三个答案全是否，通常不触发。

如果任意一个答案是“是”，通常应触发。

如果你需要例子，读取 [references/trigger-examples.md](references/trigger-examples.md)。

## 总体流程

这个 skill 的执行顺序固定如下：

1. 判断是否触发
2. 先做理解确认
3. 用户确认后按清单执行
4. 宣布完成前做硬性验收

不要跳级。

不要在还没确认理解时进入执行。

不要在还没通过验收时宣布完成。

## 第一阶段：理解确认

如果任务已被判断为复杂，先输出可见理解稿。

你必须写出：

- `我的理解`
- `你真正要的结果`
- `我当前的关键假设`
- `最容易跑偏的地方`

不要暴露冗长思维链。只写用户需要核对方向的最小必要信息。

然后给出 2 到 3 个真正影响结果的关键选项。不要让用户自己从零设计流程。

可用选项包括：

- 输出风格：简洁 / 平衡 / 详细
- 执行策略：稳妥 / 标准 / 速度优先
- 交付形式：直接结果 / 表格 / 文件 / 先方案后执行
- 优先目标：准确 / 完整 / 省时间

如果默认值明显更合理，要标出推荐项。

接着写出执行合同，必须包括：

- `Goal`
- `Deliverable`
- `Inputs`
- `Constraints`
- `Non-goals`
- `Completion gate`

要求如下：

- `Goal` 只能有一个主目标
- `Deliverable` 必须可观察
- `Constraints` 只保留硬约束
- `Non-goals` 要写清不做什么
- `Completion gate` 必须能验证

最后写出步骤清单，把任务拆成 `1..N` 个必要步骤。

每一步都必须：

- 足够具体
- 可以判断完成与否
- 尽量只承担一个动作目的

步骤状态只能使用：

- `pending`
- `in_progress`
- `done`

如果有分支条件，必须在这里提前写出。

最后请求一次性确认。推荐类似：

`如果以上理解和方案正确，我将按这个步骤清单持续执行到完成，中间不再为普通步骤逐一请求批准。`

在用户确认前，不得进入正式执行。

## 第二阶段：按清单执行

用户确认后，进入执行阶段。

核心原则：

`用户确认一次总方案后，你要自己盯着计划单推进到完成。`

执行前，先重述：

- `Goal`
- `Deliverable`
- `Constraints`
- `Completion gate`
- 当前步骤清单

这一步不是再次向用户确认，而是给执行阶段锁定目标。

每一步执行前都要写清：

- 当前步骤编号
- 当前步骤名称
- 本步目标
- 准备执行的动作或工具
- 预期结果

每一步执行后都要写清：

- 实际发生了什么
- 当前步骤是否完成
- 状态如何更新
- 下一步是什么

每完成一步，都要回头检查整张步骤清单：

- 是否还有 `pending`
- 是否还有 `in_progress`
- 是否有依赖关系尚未满足

只要还有必要步骤未完成，就继续推进。

不要在普通步骤之间反复请求用户批准。

不要问：

- “要不要继续下一步”
- “我先停在这里可以吗”
- “做到这里是否够了”

这些都属于错误的中断。

只有以下情况可以重新中断用户：

- 关键信息缺失，无法继续
- 用户原始要求与当前证据冲突
- 任务边界发生实质变化
- 执行动作涉及高风险、不可逆后果

## 第三阶段：完成前验收

在准备说“已完成”之前，必须先做验收。

逐项检查：

1. 是否所有必要步骤都已经是 `done`
2. 最终交付物是否符合用户要求的形式
3. 是否还有任何硬性约束未满足
4. 是否还存在 blocker、遗留问题、未处理分支

如果任意一项未通过，就不能宣布完成。

这时只能做两件事之一：

- 继续执行
- 明确说明 blocker

## 推荐输出结构

复杂任务推荐按两个阶段展示：

### 执行前

- `我的理解`
- `关键选择`
- `执行合同`
- `步骤清单`
- `请确认或修改`

### 用户确认后

- `任务重述`
- `执行步骤`
- `当前步骤`
- `步骤结果`
- `完成检查`

## 红旗信号

如果你脑中出现以下想法，说明你在绕流程：

- “这个任务很简单，不需要确认”
- “我先做一点再说”
- “先调一次工具看看”
- “做到这里差不多了”
- “用户应该能接受这个格式”
- “先说完成，细节之后补”

一旦出现这些信号，停止即兴发挥，回到流程。
