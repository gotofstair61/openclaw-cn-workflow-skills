# OpenClaw 中文流程 Skill Triggering Tests

这份文件模仿 `superpowers/tests/skill-triggering` 的思路，用来测试这套中文流程 skills 的触发是否正确。

测试重点不是回答质量，而是：

- 是否在复杂任务上正确触发流程
- 是否在简单任务上正确跳过流程
- 是否在边界任务上做出合理判断

---

## 测试方法

每条样例都包含：

- `编号`
- `用户输入`
- `预期结果`
- `应触发的 skill`
- `不应触发的 skill`
- `说明`

实际测试时，建议记录：

- 是否通过
- 实际触发了什么
- 是否存在误触发
- 是否存在漏触发

---

## A. 应触发样例

### TRIGGER-001

**用户输入**

```text
帮我研究一下这几个 AI 产品的区别，先说说你的理解，然后按步骤比较，最后整理成表格，不要漏项。
```

**预期结果**

- 应触发复杂任务流程
- 应先进入理解确认

**应触发的 skill**

- `openclaw-cn-model-guardrails`
- `openclaw-cn-confirm-task`

**不应触发的 skill**

- 无

**说明**

这是典型的多步骤、结构化输出、强调完整性的复杂任务。

### TRIGGER-002

**用户输入**

```text
这个任务比较复杂，你先把你的理解和计划写出来，我确认后你再执行。
```

**预期结果**

- 应触发复杂任务流程
- 应进入确认阶段

**应触发的 skill**

- `openclaw-cn-model-guardrails`
- `openclaw-cn-confirm-task`

**不应触发的 skill**

- `openclaw-cn-execute-plan`（在用户确认之前）

**说明**

用户已经明确要求先确认再执行。

### TRIGGER-003

**用户输入**

```text
帮我查资料、整理重点、再根据结果输出一个行动建议清单，中间不要漏步骤。
```

**预期结果**

- 应触发复杂任务流程
- 应要求步骤清单和完成前验收

**应触发的 skill**

- `openclaw-cn-model-guardrails`

**不应触发的 skill**

- 无

**说明**

任务需要持续推进，不是单轮回答。

### TRIGGER-004

**用户输入**

```text
把这个需求分析清楚，按步骤拆开，最后输出成“目标、约束、风险、建议”四栏结构，做完前自己检查一遍。
```

**预期结果**

- 应触发复杂任务流程
- 应包含完成前验收思路

**应触发的 skill**

- `openclaw-cn-model-guardrails`
- `openclaw-cn-confirm-task`

**不应触发的 skill**

- 无

**说明**

已经明确要求结构化交付和结束前检查。

### TRIGGER-005

**用户输入**

```text
你先确认一次方向，确认完以后就自己按步骤做完，不要每一步都停下来问我。
```

**预期结果**

- 应触发
- 应先确认，再连续执行

**应触发的 skill**

- `openclaw-cn-model-guardrails`
- `openclaw-cn-confirm-task`
- `openclaw-cn-execute-plan`

**不应触发的 skill**

- 无

**说明**

这是这套流程设计要解决的核心场景之一。

### TRIGGER-006

**用户输入**

```text
我要你帮我整理这个项目方案，先确认你理解得对不对，再输出最终版本。这个东西做错了返工会很多。
```

**预期结果**

- 应触发
- 应优先确认理解

**应触发的 skill**

- `openclaw-cn-model-guardrails`
- `openclaw-cn-confirm-task`

**不应触发的 skill**

- 无

**说明**

高返工成本意味着不该直接动手。

## C. 边界样例

### BORDERLINE-001

**用户输入**

```text
帮我整理这份资料。
```

**预期结果**

- 不能机械触发
- 需要先判断复杂度

**应触发的 skill**

- 视上下文而定

**不应触发的 skill**

- 无固定答案

**说明**

如果只是简单汇总，不触发；如果需要标准确认、分步骤提取、结构化输出，则触发。

### BORDERLINE-002

**用户输入**

```text
帮我写一个会议纪要模板。
```

**预期结果**

- 通常不触发
- 如需先比较方向、确认模板风格，则转为触发

**应触发的 skill**

- 视后续上下文而定

**不应触发的 skill**

- 无固定答案

**说明**

单次生成通常简单，但也可能被用户升级成复杂确认流程。

### BORDERLINE-003

**用户输入**

```text
帮我看看这个产品值不值得买。
```

**预期结果**

- 简单问法通常不触发
- 多维度比较、多来源核对时应触发

**应触发的 skill**

- 视任务展开程度而定

**不应触发的 skill**

- 无固定答案

**说明**

这类请求最容易因为上下文不同而变化。

### BORDERLINE-004

**用户输入**

```text
帮我改一下这个方案。
```

**预期结果**

- 先判断是文字修改还是过程任务

**应触发的 skill**

- 视修改深度而定

**不应触发的 skill**

- 无固定答案

**说明**

纯润色不触发；重新理解目标、重组结构、分步修订则触发。

### BORDERLINE-005

**用户输入**

```text
你帮我调研一下这个方向。
```

**预期结果**

- 不能立刻机械判断
- 需要判断是否涉及多步骤研究和结构化结论

**应触发的 skill**

- 常常会触发，但不一定

**不应触发的 skill**

- 无固定答案

**说明**

“调研”本身偏复杂，但还要看用户到底要多深。

## E. 失败信号

如果测试时出现以下情况，说明触发规则需要继续修：

- 简单翻译、润色、问答也被拉进复杂流程
- 明显复杂任务却没有触发
- 用户明确要求先确认，模型却直接执行
- 用户明确要求不要每步都问，模型仍逐步停顿
- 模型在复杂任务里没有进入完成前验收
